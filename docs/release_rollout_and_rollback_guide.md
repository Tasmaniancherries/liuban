# 留伴灰度与回滚判定指南

本指南用于发布当天快速决策：继续放量、保持观察、或立即回滚。

## 适用范围

- 版本发布日（T 日）到发布后首周（T+7）
- iOS / Android 双端

## 放量节奏（建议）

1. 阶段 1：5% 灰度，观察 2-4 小时
2. 阶段 2：20% 灰度，观察半天
3. 阶段 3：50% 灰度，观察 1 天
4. 阶段 4：100% 全量

若任一阶段触发回滚条件，停止放量并执行回滚流程。

## 关键观察指标

- 稳定性
  - 崩溃率（Crash-free users）
  - 非致命异常量（按版本）
- 核心转化
  - 登录成功率
  - 注册提交成功率
  - 发帖成功率
  - 私信发送成功率
- 用户体验
  - 关键页面首屏失败率（空态+错误提示触发频次）
  - 客服工单新增量（按问题类型）

## 回滚触发条件（参考情形）

满足任一条件即可触发回滚评估：

- 崩溃率较上一稳定版本显著恶化，且持续 30 分钟以上
- 登录/注册等关键成功率出现持续下降，影响主流程
- 大量用户集中反馈“无法使用主功能”（发帖/消息/登录）
- 服务端与客户端版本组合出现系统性兼容问题

> 说明：阈值请以你们当前监控平台与历史基线为准，发布前由产品/研发/测试共同确认。

## 回滚执行流程（简版）

1. 研发值班宣布“暂停放量”，锁定当前版本
2. 通知产品、测试、运营、客服进入回滚状态
3. 平台侧执行版本回退（或停止继续放量）
4. 客服使用统一口径回复用户（已知问题 + 预计恢复时间）
5. 研发产出事故记录：影响范围、根因、修复计划、复发防护

## 发布当天角色分工

- 研发：监控看板、异常定位、回滚执行
- 测试：关键路径复测、回归确认
- 产品：放量决策与风险权衡
- 运营/客服：用户沟通与问题收集
