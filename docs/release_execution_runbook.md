# 留伴发布执行顺序（Runbook）

本指引用于在发布日前后快速对齐产品、测试、研发、运营的动作顺序。

## T-3 ~ T-1（发布前准备）

- 负责人：研发 + 测试
- 使用文档：`release_readiness_checklist.md`
- 目标：完成 P0 阻塞项，确认可发布

建议顺序：

1. 双平台真机回归（主路径全走通）
2. 后端联调签收（成功/失败/空态/超时）
3. 崩溃与错误监控接入验证
4. 发布构建链路验证（版本号、签名、产物）
5. 隐私与合规入口核对

## T-1（发布评审）

- 负责人：产品 + 研发 + 测试 + 运营
- 使用文档：`release_notes_launch_candidate.md`
- 目标：统一内部口径，确认灰度方案与回滚阈值

评审建议：

1. 过一遍「本版重点」与「用户可感知变更」
2. 确认风险与关注点是否可接受
3. 明确灰度比例、观察窗口、回滚条件与负责人

## T（日发布）

- 负责人：研发 + 运营
- 使用文档：`release_notes_public_short.md`
- 目标：对外发布文案一致、简洁、可直接投放

发布建议：

1. 商店更新说明使用对外简版文案
2. 社群公告保持同口径，避免技术实现细节
3. 客服同步 FAQ（网络异常重试、入口路径）

## T+1 ~ T+7（发布后观察）

- 负责人：研发 + 运营 + 客服
- 使用文档：`release_readiness_checklist.md`（P2）
- 目标：确认指标、收敛问题、安排优化

建议动作：

1. 每日查看崩溃率与关键路径成功率
2. 汇总高频用户反馈并按优先级建单
3. 周内形成首轮复盘与后续排期
